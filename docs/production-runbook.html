Runbook de salida a producción

Versión inicial: 21/07/2026. Ejecutar en orden y guardar evidencia (PR, resultado CI y captura/configuración) de cada paso.

0. Regla de bloqueo

No desplegar hasta que la migración de seguridad esté aplicada al proyecto Supabase correcto, los secretos estén rotados y CI esté verde. Esta aplicación conecta desde navegador a Supabase: RLS y RPC son el perímetro de seguridad.

1. Contención inmediata

  1. Rotar SUPABASE_ACCESS_TOKEN en el panel de Supabase. Revocar el valor anterior y revisar accesos/logs.
  2. Quitar .env del control de versiones y mantener sólo .env.example; configurar valores reales exclusivamente en Vercel/Supabase/GitHub Secrets.
  3. Revisar todos los perfiles con rol admin. Mantener únicamente administradores autorizados.
  4. Desplegar la migración de hardening antes de habilitar nuevos registros o reservas.

2. Reconciliar la base real

  1. Generar un dump de esquema de la instancia Supabase de producción (tablas, vistas, funciones, policies, grants y triggers).
  2. Compararlo con supabase/migrations. Versionar las RPC usadas por el frontend: aprobación, métricas, reservas por dueño y onboarding.
  3. Aplicar las migraciones en un proyecto Supabase nuevo de staging. Nunca probar una migración nueva por primera vez contra producción.
  4. Regenerar src/integrations/supabase/types.ts desde el esquema reconciliado y eliminar los casts any que oculten incompatibilidades.

3. Variables por entorno

  • Frontend/Vercel: VITE_SUPABASE_URL, VITE_SUPABASE_PUBLISHABLE_KEY y el ID de proyecto si se usa. La publishable key puede estar en cliente; un service-role/token personal no.
  • Supabase Edge Functions: sólo secretos del servidor mediante supabase secrets set (service role, proveedores de pago/notificación).
  • Ambientes: proyectos independientes para development, staging y production; OAuth redirect URLs y CORS específicos por ambiente.

4. Validación funcional en staging

  1. Cliente: registro, login/logout, edición de campos permitidos, catálogo, detalle, reserva válida, doble reserva concurrente rechazada y cancelación permitida.
  2. Propietario: onboarding, alta/edición sólo de su complejo, visualización de sus reservas, sin capacidad de alterar pagos confirmados.
  3. Administrador: un admin preasignado puede aprobar; un cliente/owner no puede acceder ni llamar RPC administrativa.
  4. Seguridad: intentar cambiar role, precio, user_id, cancha ajena, complejo ajeno y estado de pago directamente desde la API; todo debe ser denegado.
  5. Notificaciones: Edge Function sin JWT, con cuerpo inválido y desde origen no permitido deben fallar. Una reserva válida debe producir sólo una notificación autorizada.

5. Gate automatizado

En una copia limpia del repositorio ejecutar:

npm ci
npm run typecheck
npm run lint
npm run build

GitHub Actions ejecuta estas validaciones para PR y main. Añadir antes del lanzamiento pruebas de integración de RLS/RPC y pruebas E2E del flujo de reserva.

6. Despliegue

  1. Abrir PR con migración, frontend y documentación; requerir revisión y CI verde.
  2. Aplicar migración primero a staging, ejecutar el paso 4 y promover a producción en una ventana controlada.
  3. Configurar variables de producción en Vercel; publicar desde main sólo después del gate.
  4. Hacer smoke test público: carga SPA, autenticación, lectura de catálogo, reserva y paneles autorizados.
  5. Monitorear durante 24 horas: errores de Edge Functions, rechazos de RPC, creación de reservas, roles y pagos.

7. Rollback

Preparar una migración compensatoria antes de aplicar cambios de esquema. Para frontend, volver al deployment anterior de Vercel. Para secretos comprometidos no hay rollback: se revocan y rotan. No deshabilitar RLS como medida de emergencia.

Salida aprobada sólo con: roles auditados, migración aplicada, secretos rotados, tests verdes, smoke test aprobado y monitoreo activo.
